    AgentMesh CLI 使用指南¶

    AgentMesh CLI 是基于 A2A (Agent-to-Agent) 协议的 Agent 编排命令行工具，提供项目初始化、Schema 验证、消息格式转换、保真度追踪和流程执行能力。

    环境要求¶

    项目 要求
    Python >= 3.10
    Click >= 8.1 (自动安装)
    A2A SDK 需提供有效路径 (见下文)

    安装¶

    方式一：pip 安装¶

    cd agentmesh/cli
pip install .

    方式二：开发模式安装¶

    pip install -e ".[dev]"

    方式三：直接运行¶

    python -m agentmesh --help

    安装后验证：

    agentmesh --version
# 输出: agentmesh, version 0.1.0

agentmesh --help
# 输出 5 个命令清单

    全局选项¶

    所有命令共享以下两个全局选项：

    选项 简写 说明
    --sdk-path (无) A2A SDK 目录路径，也支持环境变量 AGENTMESH_SDK_PATH
    --verbose -v 输出详细信息，显示每个命令的额外分析数据

    SDK 路径解析¶

    CLI 依赖 A2A SDK (a2a_adapter.py + a2a_models.py)。SDK 路径按以下优先级查找：

    1. --sdk-path 参数 — 最高优先级
    2. AGENTMESH_SDK_PATH 环境变量
    3. 默认路径列表（按顺序尝试）:
    4. ~/.openclaw/workspace/zhongshuling/dev/active/agentmesh/sdk/
    5. CLI 所在目录的同级 sdk/ 目录
    6. 当前工作目录下的 sdk/ 目录

    如果 SDK 找不到，CLI 会报错退出。建议通过环境变量固定 SDK 路径：

    export AGENTMESH_SDK_PATH=/path/to/a2a-sdk

    命令 1: init — 初始化项目¶

    创建 AgentMesh 项目骨架目录和初始配置文件。

    用法¶

    agentmesh init [OPTIONS] [PROJECT_NAME]

    参数¶

    参数 说明
    PROJECT_NAME 项目目录名，默认 agentmesh-project，可选

    选项¶

    选项 简写 说明
    --force -f 如果目标目录已存在则覆盖
    --sdk-path (无) SDK 路径
    --verbose -v 输出目录结构详情

    生成结构¶

    <project-name>/
  agentmesh.yaml         # 项目配置文件 (JSON)
  agents/                # Agent 定义文件目录
  workflows/             # 工作流定义文件目录
  config/                # 配置目录
  data/
    input/               # 输入数据目录
    output/              # 输出数据目录
    fidelity/            # 保真度报告目录
    logs/                # 日志目录

    示例¶

    示例 1: 创建一个默认项目

    agentmesh init my-first-project

    输出:

    AgentMesh 项目 'my-first-project' 已初始化:
  位置: /current/working/dir/my-first-project
  目录数: 7
  配置: agentmesh.yaml

    生成的 agentmesh.yaml:

    {
  "project": "my-first-project",
  "version": "0.1.0",
  "schema_version": "2.1",
  "agents": {
    "path": "agents/",
    "agent_cards": {}
  },
  "workflows": {
    "path": "workflows/"
  },
  "a2a": {
    "protocol_version": "1.0.0",
    "agent_card_url": "http://localhost:8000/.well-known/agent.json"
  },
  "fidelity": {
    "tracking": true,
    "threshold": 0.7,
    "report_path": "data/fidelity/"
  }
}

    示例 2: 覆盖已有目录并查看详细结构

    agentmesh init my-project --force -v

    输出:

    AgentMesh 项目 'my-project' 已初始化:
  位置: /current/working/dir/my-project
  目录数: 7
  配置: agentmesh.yaml

目录结构:
  agents/
  workflows/
  config/
  data/input/
  data/output/
  data/fidelity/
  data/logs/

    命令 2: validate — 验证 Schema¶

    验证 JSON 文件是否符合 A2A Schemas（Task、Message、AgentCard、Request、Response）。

    用法¶

    agentmesh validate [OPTIONS] INPUT_FILE

    参数¶

    参数 说明
    INPUT_FILE 要验证的 JSON 文件路径，必填

    选项¶

    选项 简写 说明
    --schema (无) Schema 类型：task (默认), message, card, request, response
    --sdk-path (无) SDK 路径
    --verbose -v 输出解析摘要

    示例¶

    示例 1: 验证 A2A Task

    输入文件 task.json:

    {
  "id": "task-001",
  "status": {
    "state": "completed"
  },
  "history": [
    {
      "message_id": "msg-1",
      "role": "user",
      "parts": [
        {"type": "text", "text": "Hello"}
      ]
    }
  ]
}

    命令:

    agentmesh validate task.json --schema task

    输出:

    验证通过: A2ATask 格式正确

    示例 2: 验证 AgentCard 并查看摘要

    输入文件 card.json:

    {
  "name": "Weather Agent",
  "description": "提供天气查询服务",
  "url": "http://weather-agent.local",
  "skills": [
    {
      "id": "get-weather",
      "name": "获取天气",
      "description": "根据城市名查询实时天气",
      "inputs": [
        {"name": "city", "type": "string", "description": "城市名称"}
      ],
      "outputs": [
        {"name": "temperature", "type": "number", "description": "温度"}
      ]
    }
  ]
}

    命令:

    agentmesh validate card.json --schema card -v

    输出:

    验证通过: AgentCard 格式正确

摘要: {
  "name": "Weather Agent",
  "description": "提供天气查询服务",
  "url": "http://weather-agent.local",
  "skills_count": 1
}

    命令 3: convert — 格式转换¶

    在 AgentMesh 内部消息格式和 A2A 标准格式之间双向转换。

    用法¶

    agentmesh convert [OPTIONS] INPUT_FILE

    参数¶

    参数 说明
    INPUT_FILE 要转换的 JSON 文件，必填

    选项¶

    选项 简写 说明
    --direction -d 转换方向：to-a2a, from-a2a, auto (默认，自动判断)
    --output -o 输出文件路径 (默认输出到终端)
    --pretty -p 美化 JSON 输出
    --sdk-path (无) SDK 路径
    --verbose -v 输出转换摘要

    自动方向判断规则：

    • 输入包含 message_type 字段 → to-a2a (AgentMesh -> A2A)
    • 输入包含 id + status 或 jsonrpc / method → from-a2a (A2A -> AgentMesh)

    示例¶

    示例 1: AgentMesh 消息转 A2A Task

    输入文件 agentmesh-msg.json:

    {
  "message_type": "task_request",
  "sender": "user-001",
  "target_agent": "weather-agent",
  "payload": {
    "task": "query_weather",
    "parameters": {
      "city": "Beijing"
    },
    "description": "查询北京天气"
  },
  "metadata": {
    "priority": "high"
  }
}

    命令:

    agentmesh convert agentmesh-msg.json --direction to-a2a -p

    输出:

    {
  "direction": "to-a2a",
  "agentmesh_message_type": "task_request",
  "a2a": {
    "task": {
      "id": "<uuid>",
      "status": {
        "state": "submitted"
      },
      "history": [
        {
          "message_id": "<uuid>",
          "role": "user",
          "parts": [
            {
              "type": "text",
              "text": "查询北京天气"
            },
            {
              "type": "data",
              "data": {
                "city": "Beijing",
                "priority": "high"
              }
            }
          ]
        }
      ],
      "metadata": {
        "source": "agentmesh-cli",
        "original_type": "task_request",
        "original_sender": "user-001",
        "original_target": "weather-agent"
      }
    },
    "jsonrpc_request": {
      "jsonrpc": "2.0",
      "method": "tasks/send",
      "params": {
        "id": "<uuid>"
      },
      "id": "1"
    }
  }
}

    示例 2: A2A Task 转 AgentMesh 消息

    输入文件 a2a-task.json:

    {
  "id": "task-abc-123",
  "status": {
    "state": "completed",
    "timestamp": "2026-05-22T10:00:00Z"
  },
  "history": [
    {
      "message_id": "msg-001",
      "role": "agent",
      "parts": [
        {
          "type": "text",
          "text": "北京当前气温 26°C，晴朗"
        },
        {
          "type": "data",
          "data": {
            "city": "Beijing",
            "temperature": 26,
            "condition": "sunny"
          }
        }
      ]
    }
  ],
  "artifacts": [
    {
      "name": "weather-result",
      "parts": [
        {
          "type": "text",
          "text": "完整天气报告已生成"
        }
      ]
    }
  ]
}

    命令:

    agentmesh convert a2a-task.json --direction from-a2a -p

    输出:

    {
  "direction": "from-a2a",
  "agentmesh": {
    "message_type": "task_response",
    "sender": "agent",
    "content": "北京当前气温 26°C，晴朗",
    "data": {
      "city": "Beijing",
      "temperature": 26,
      "condition": "sunny"
    },
    "artifacts": [
      {
        "name": "weather-result",
        "content": "完整天气报告已生成"
      }
    ],
    "task_id": "task-abc-123",
    "state": "completed"
  }
}

    命令 4: fidelity — 保真度追踪¶

    比较输入和输出消息之间的信息保留率，生成多维度的保真度评分报告。

    用法¶

    agentmesh fidelity [OPTIONS]

    选项¶

    选项 简写 说明
    --input -i 输入消息 JSON 文件，必填
    --output -o 输出消息 JSON 文件，必填
    --report -r 保真度报告输出路径 (默认输出到终端)
    --sdk-path (无) SDK 路径
    --verbose -v 输出详细报告（含缩进JSON）

    保真度维度说明¶

    维度 权重 说明
    state_preserved 15% 状态一致性 (输入/输出的 state 值是否相同)
    text_similarity 35% 文本内容 Jaccard 相似度 (单词集合重叠率)
    data_preserved 25% 数据字段保留率
    artifact_preserved 15% Artifact 数量保留率
    metadata_preserved 10% 元数据字段保留率

    综合评分低于 0.7 (70%) 时输出警告。

    示例¶

    示例 1: 同任务保真度评估

    输入文件 input.json:

    {
  "message_type": "task_request",
  "sender": "user",
  "payload": {
    "task": "query_weather",
    "parameters": {"city": "Beijing"},
    "description": "查询北京天气"
  }
}

    输出文件 output.json:

    {
  "message_type": "task_response",
  "sender": "agent",
  "content": "北京天气：26°C，晴朗",
  "data": {"city": "Beijing", "temperature": 26}
}

    命令:

    agentmesh fidelity --input input.json --output output.json

    输出:

    {"state_preserved": 0.0, "input_state": "submitted", "output_state": "completed", "text_similarity": 0.3333, "data_preserved": 0.5, "artifact_preserved": 1.0, "metadata_preserved": 1.0, "overall_fidelity": 0.6, "input_message_type": "task_request", "output_message_type": "task_response"}

    示例 2: 高保真转换评估并写入报告文件

    使用 -v 可读输出：

    agentmesh fidelity --input input.json --output output.json -v -r fidelity-report.json

    输出:

    保真度报告已写入: fidelity-report.json

保真度分析详情:
  输入: input.json
  输出: output.json
  Task ID: <uuid>

警告: 保真度 60% 低于阈值 70%

    生成的 fidelity-report.json:

    {"state_preserved": 0.0, "input_state": "submitted", "output_state": "completed", "text_similarity": 0.3333, "data_preserved": 0.5, "artifact_preserved": 1.0, "metadata_preserved": 1.0, "overall_fidelity": 0.6, "input_message_type": "task_request", "output_message_type": "task_response"}

    命令 5: run — 流程执行¶

    加载工作流定义 JSON 文件，解析执行步骤并模拟执行。

    用法¶

    agentmesh run [OPTIONS] WORKFLOW_FILE

    参数¶

    参数 说明
    WORKFLOW_FILE 工作流定义 JSON 文件，必填

    选项¶

    选项 简写 说明
    --agent-card -c Agent Card JSON 文件 (可选)
    --dry-run (无) 仅解析并输出执行计划，不实际执行
    --sdk-path (无) SDK 路径
    --verbose -v 输出详细信息

    工作流文件格式¶

    {
  "name": "工作流名称",
  "version": "1.0",
  "steps": [
    {
      "message_type": "task_request",
      "sender": "user",
      "payload": {
        "task": "query_weather",
        "description": "查询北京天气"
      }
    },
    {
      "message_type": "task_response",
      "sender": "weather-agent",
      "payload": {
        "description": "返回天气结果"
      }
    }
  ]
}

    支持两种字段名： - steps — 推荐，用于命名清晰的步骤列表 - messages — 兼容旧格式

    示例¶

    示例 1: 查看工作流执行计划 (dry-run)

    输入文件 weather-workflow.json:

    {
  "name": "天气查询流程",
  "version": "1.0",
  "steps": [
    {
      "message_type": "task_request",
      "sender": "user",
      "payload": {
        "task": "query_weather",
        "description": "查询北京天气"
      }
    },
    {
      "message_type": "task_response",
      "sender": "weather-agent",
      "payload": {
        "task": "response",
        "description": "返回北京的天气结果，包括温度和天气状况"
      }
    }
  ]
}

    命令:

    agentmesh run weather-workflow.json --dry-run

    输出:

    工作流: 天气查询流程 (v1.0)
步骤数: 2

--- 执行计划 (DRY RUN) ---
  步骤 1: task_request from user
          查询北京天气
  步骤 2: task_response from weather-agent
          返回北京的天气结果，包括温度和天气状况

模拟执行完成 (dry-run, 未实际执行)

    示例 2: 加载 Agent Card 执行工作流

    agentmesh run weather-workflow.json --agent-card agent-card.json -v

    输出:

    已加载 Agent Card: Weather Agent
工作流: 天气查询流程 (v1.0)
步骤数: 2

开始执行...
  [1/2] task_request...
    -> 处理完成 (占位)
  [2/2] task_response...
    -> 处理完成 (占位)

工作流 '天气查询流程' 执行完成
处理步骤: 2 个

    常见问题¶

    Q: 提示 "未找到 A2A SDK" 怎么办？¶

    CLI 依赖 A2A SDK (a2a_adapter.py + a2a_models.py)。设置环境变量或在命令中指定 SDK 路径：

    export AGENTMESH_SDK_PATH=/path/to/your/sdk
agentmesh validate task.json

    或者每次使用时加 --sdk-path：

    agentmesh validate task.json --sdk-path /path/to/your/sdk

    Q: validate 验证失败，但 JSON 看起来没问题？¶

    检查 --schema 参数是否正确。五种 Schema 各有严格的结构要求：

    • task — 需要 id + status.state + history
    • message — 需要 message_id + role + parts
    • card — 需要 name + description + url + skills
    • request — 需要 method + id
    • response — 需要 id + result 或 error

    Q: convert 自动判断方向错误？¶

    自动判断规则的局限： - 包含 message_type 字段 → 判为 AgentMesh -> A2A - 否则判为 A2A -> AgentMesh

    如果输入文件同时包含两种风格的特征，手动指定方向：

    agentmesh convert ambiguous.json --direction to-a2a

    Q: fidelity 评分总是很低？¶

    保真度评分（特别是 state_preserved）要求输入和输出消息的类型相近。不同类型（如 task_request -> task_response）状态不同，state_preserved 自然为 0，属于正常现象。重点关注 text_similarity 和 data_preserved 维度。

    Q: run 命令实际执行了什么？¶

    当前版本 run 命令会加载工作流定义、解析 Agent Card 并逐步输出处理状态，但实际业务执行逻辑为占位实现，主要用于验证工作流定义的正确性和执行流程的完整性。

    快速参考¶

    # 帮助
agentmesh --help
agentmesh <command> --help

# 初始化项目
agentmesh init my-project
agentmesh init my-project --force -v

# 验证
agentmesh validate task.json
agentmesh validate card.json --schema card -v

# 转换
agentmesh convert msg.json --direction to-a2a -p
agentmesh convert a2a-task.json --direction from-a2a -o result.json

# 保真度
agentmesh fidelity -i input.json -o output.json -v

# 运行
agentmesh run workflow.json --dry-run
